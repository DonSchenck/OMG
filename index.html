Problem #1: What's in a name?

Naming your project is a challenge. Is the name already being used? Does the name you choose directly relate to your project? Is it a "cute" name that will leave folks confused. This is not easy; give it a lot of thought.

Problem #2: Where to put it?

Where will you put your project? If it's YOUR project, feel free to put it in your own Github account. But if it's a part of your job, you might want to put it in your company's account. If your workplace doesn't have a Github account, guess what: You have the opportunity to start something!
Just make sure it's in a place where it will be safe and seen for a long time.

github.com/DonSchenck/PoshStack

Of course, I chose my own account ... which was a mistake; the code was directly related to OpenStack, a Rackspace project. I SHOULD have put it in the Rackspace account.
So there's a good mistake to avoid.

Get ready for some *serious* constructive criticism. Don't be thinned-skinned; put on the fire suit and get ready. Take EVERYTHING as a POSITIVE intention.

I TOLD YOU This is a public project!

YOU don't own this; it's a *public* project. It's open to public discussion and criticism and...

...public bug reporting, and...

...public HELP! (yay)

"We accept Pull Requests"

Remember this phrase if anyone is overly critical of your project. After all, they're free to contribute. Don't let anyone make you feel like YOU must support THEIR needs.

Help is on the way

People will start to take an interest. It may start as emails, which YOU need to turn into Issues. Encourage folks to CREATE ISSUES. Don't forget where you need help: NOT just code ... documentation ... tests ... sample code. Well-intentioned people may show interest but never show up to help.

The lull

Nothing. Crickets. This is when you bear down and start The Work. You write code, you write documentation, you write sample code, you keep publicizing the effort and asking for help. DO. NOT. QUIT.
